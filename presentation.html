The Performance Challenge vs. Compiler Objective

🚨 THE SQL PERFORMANCE PROBLEM

Why database queries execute sub-optimally:

  • Redundant Conditions: Frameworks and programmers generate dead conditions like WHERE 1=1 AND status = 'A', wasting parser/evaluator cycles.
  • Delayed Filtering: Applying table filters *after* joins forces databases to build massive intermediate cross-product relations in memory.
  • Opaque Execution Plans: Relational engines perform optimizations hidden from developers, offering no visibility into the parsing or AST structures.
✓ THE COMPILER-DRIVEN SOLUTION

How our optimization framework resolves these gaps:

  • AST Simplification: Translates statements into an AST, resolving and folding constant conditions prior to database execution.
  • Early Join Filtering: Pushes filter conditions directly into the JOIN ON criteria, shrinking joining relation sizes.
  • Full Visualization: Exposes the complete compiler pipeline—from raw source and tokens to the AST and optimized query outputs.

Optimization Pass 1: Constant Folding

Pre-evaluates comparisons containing only constants and simplifies logic gates recursively:

Original Condition Tree
Condition(AND)
Condition(COMP)
left: Column(salary), op: '>', right: Constant(50000)
Condition(COMP)
left: Constant(1), op: '=', right: Constant(1)
Query: ... WHERE salary > 50000 AND 1 = 1
Folding Engine
➔
1 = 1 evaluates to TRUE
➔
X AND TRUE simplies to X
Folded Condition Tree
Condition(COMP)
left: Column(salary), op: '>', right: Constant(50000)
Query: ... WHERE salary > 50000
AND Simplification FALSE AND X ➔ FALSE
TRUE AND X ➔ X
OR Simplification TRUE OR X ➔ TRUE
FALSE OR X ➔ X
WHERE Removal If WHERE clause folds entirely to TRUE, the WHERE condition is removed from the query AST.

Optimization Pass 2: Predicate Pushdown

Analyzes compound condition lists and moves filter expressions directly into matching JOIN ON clauses:

Before: Filtering After Joining 
SELECT e.name, d.dept_name
FROM employees e
JOIN departments d ON e.dept_id = d.id
WHERE d.dept_name = 'Engineering'
Execution path:
  1. Joins all rows of employees and departments (large output).
  2. Filters the huge joined table by dept_name.
After: Pushed Predicate 
SELECT e.name, d.dept_name
FROM employees e
JOIN (
  SELECT * FROM departments 
  WHERE dept_name = 'Engineering'
) d ON e.dept_id = d.id
Execution path:
  1. Filters departments table rows first during the join process.
  2. Performs join only on matching rows, shrinking intermediate sets.
Why it works:

Relational Algebra equivalence: σ_A(R ⋈_θ S) ➔ R ⋈_θ (σ_A(S)) when condition A only references attributes in relation S. By pushing the selection predicate down, we reduce the cardinality of relations being joined, yielding substantial query speedups.

Full-Stack Integration & Benchmarking Engine
